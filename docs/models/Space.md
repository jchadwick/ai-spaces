# Model: Space

**Purpose:** Space metadata

**Storage:** Database `spaces` table

---

## Zod Schema

```typescript
import { z } from "zod";

export const SpaceSchema = z.object({
  id: z.string().uuid(),
  name: z.string().min(1).max(100),
  description: z.string().max(500).optional(),
  ownerId: z.string().uuid(),
  createdAt: z.string().datetime(),
  updatedAt: z.string().datetime(),
});

export type Space = z.infer<typeof SpaceSchema>;
```

---

## Fields

### `id`

- **Type:** `UUID`
- **Description:** Unique identifier for this space
- **Generated by:** Spaces Service database
- **Used by:** Agent Adapter to map to specific agent and location

### `name`

- **Type:** `string` (1-100 characters)
- **Description:** Human-readable name for the space
- **Source:** Agent's `.space/spaces.json` (read via adapter)

### `description`

- **Type:** `string` (0-500 characters, optional)
- **Description:** Brief description of the space
- **Source:** Agent's `.space/spaces.json` (read via adapter)

### `ownerId`

- **Type:** `UUID`
- **Description:** User who owns this space
- **References:** `users.id`
- **Permissions:** Owner has full control (can create/revoke shares)

### `createdAt`

- **Type:** `Date` (ISO 8601)
- **Description:** When the space was registered

### `updatedAt`

- **Type:** `Date` (ISO 8601)
- **Description:** When the space metadata was last updated

## Architecture: Who Knows What

### Spaces Service Knows:
- Space metadata (id, name, description, owner)
- User accounts and authentication
- Share links and permissions

### Agent Adapter Knows:
- Mapping: `spaceId` → `agentType` + `agentInstanceId` + `location`
- How to communicate with each agent type
- How to resolve file paths relative to space

### Agent Knows:
- Its own files and workspace
- Space config (`.space/spaces.json`)
- Space context (`.space/SPACE.md`)

## Lifecycle

### Registration Flow

```
Agent Owner                Agent Adapter               Spaces Service
     │                          │                             │
     │ 1. Create .space/        │                             │
     │    in workspace          │                             │
     │                          │                             │
     │ 2. Create space          │                             │
     │    POST /api/spaces       │                             │
     │    { name, description } │                             │
     │ ─────────────────────────────────────────────────────►│
     │                          │                             │
     │                          │    3. Create space record    │
     │                          │       Generate spaceId        │
     │                          │                             │
     │                          │    4. Return spaceId          │
     │ ◄──────────────────────────────────────────────────── │
     │                          │                             │
     │ 5. Register mapping       │                             │
     │    (agent adapter internal)│                           │
     │                          │                             │
```

### Agent Adapter Internal Mapping

The agent adapter maintains an internal mapping (implementation-specific):

```typescript
// OpenClaw Adapter mapping
{
  "550e8400-...": {
    agentType: "openclaw",
    agentInstanceId: "my-openclaw",
    workspacePath: "/home/user/.openclaw/workspace",
    spacePath: "Vacations"
  }
}

// Future: Other agent types might have different location formats
{
  "660e8400-...": {
    agentType: "claude-code",
    agentInstanceId: "work-claude",
    location: { /* agent-specific */ }
  }

```

## File Paths

All file paths in API calls are **relative to the space**:

```
# Spaces Service calls adapter:
getFile(spaceId: "550e8400-...", path: "Maine.md")

# Agent Adapter resolves to agent:
# OpenClaw: /home/user/.openclaw/workspace/Vacations/Maine.md
# Other agent: depends on agent type
```

## Database Schema

```sql
CREATE TABLE spaces (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name TEXT NOT NULL,
    description TEXT,
    owner_id UUID REFERENCES users(id),
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_spaces_owner ON spaces(owner_id);
```

---

## Config Not Stored in Database

**Important:** Space config (tools, capabilities) stays in `.space/spaces.json` in the agent's workspace.

Why?
1. Config is agent-specific (different agents may have different config formats)
2. Config travels with the space (if you move/copy the folder, config moves with it)
3. Spaces Service doesn't need to interpret config (agent adapter does that)

When Spaces Service needs config info, it asks the agent adapter:
```
Spaces Service → Agent Adapter: "Get space config for spaceId X"
Agent Adapter → Agent: "Read .space/spaces.json"
Agent Adapter → Spaces Service: { name, description, agent: {...} }
```

## API Endpoints

```
POST   /api/spaces              Create space
GET    /api/spaces/{id}         Get space details
GET    /api/spaces/{id}/config  Get space config (via agent adapter)
DELETE /api/spaces/{id}         Delete space
```
